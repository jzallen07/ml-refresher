    # Understanding `requires_grad=True`
    
    ## What It Does
    
    `requires_grad=True` tells PyTorch: *"Track every operation performed on this tensor so I can compute gradients later."*
    
    Think of it as turning on a "recording mode" for a tensor. Any math you do with it gets recorded in a **computational graph** — a chain of operations that PyTorch can trace backward to compute derivatives.
    
    ## Why We Need It
    
    Neural networks learn via **gradient descent**:
    1. Make a prediction
    2. Calculate how wrong the prediction is (loss)
    3. Compute how each weight contributed to that error (gradients)
    4. Adjust weights to reduce the error
    
    Steps 2-3 require gradients. Without `requires_grad=True`, PyTorch has no way to know which tensors need gradient computation.
    
    ## Simple Example
    
    ```python
    import torch
    
    # Without requires_grad - PyTorch doesn't track this
    a = torch.tensor(3.0)
    b = a ** 2
    print(b)  # tensor(9.) - just a number, no gradient info
    
    # With requires_grad - PyTorch tracks operations
    x = torch.tensor(2.0, requires_grad=True)
    y = x ** 2  # y = x²
    
    # This builds a computational graph:
    # x → (power of 2) → y
    # PyTorch knows: "to compute dy/dx, use the chain rule"
    
    y.backward()  # Compute the gradient
    
    print(x.grad)  # tensor(4.) → dy/dx = 2x = 2(2) = 4
    ```
    
    ## The Key Insight
    
    | `requires_grad=False` (default) | `requires_grad=True` |
    |--------------------------------|----------------------|
    | Just a regular number/tensor | Tracks all operations |
    | No memory of how it was created | Builds a computational graph |
    | Can't compute gradients | Can compute gradients via `.backward()` |
    
    ## When to Use It
    
    - **Yes**: On your model weights and inputs during training
    - **No**: On your input data or during inference (use `torch.no_grad()` to disable it)
    
    This automatic tracking is what makes PyTorch powerful — you don't need to manually compute derivatives; PyTorch does it for you automatically!

    ```python
    print(x.grad)                              # tensor(6.)   (derivative of x² is 2x)
    ```
    
    Without `requires_grad=True` PyTorch skips building the computational graph, so memory is saved and computation is faster—exactly what you want when you’re done training and just using the network for predictions.

    ```python
    import torch

    a = torch.tensor(3.0)

    print(b.grad_fn) # None — PyTorch didn't track anything

    x = torch.tensor(3.0, requires_grad=True)

    print(x.grad)    # tensor(5.) — the derivative of 5x is 5
    ```

    When you set `requires_grad=True`, PyTorch secretly builds a **computational graph** as you do math:
    
    ```
    x (3.0)  ──→  [ multiply by 5 ]  ──→  y (15.0)
    ```
    
    When you call `y.backward()`, PyTorch walks this graph **in reverse** and asks:
    
    > "If I nudge `x` a tiny bit, how much does `y` change?"
    
    For `y = 5x`, the answer is **5** — and that's exactly what shows up in `x.grad`.
    
    ### The Key Intuition
    
    - **Without** `requires_grad=True`: PyTorch is a calculator. It gives you answers.
    - **With** `requires_grad=True`: PyTorch is a calculator **with a memory**. It gives you answers *and* remembers how it got them, so it can compute derivatives later.
    
    In training, the model's **weights** are the tensors that get `requires_grad=True`, because those are the values we want to adjust based on the gradients.

    So `x.grad` will be `tensor(4.)`.
    
    ---
    
    ## Step 3: What PyTorch actually does internally
    
    When you write `z = x**2 + 3*y`, PyTorch builds this graph:
    
    ```
    x ──→ [x**2] ──→ [+] ──→ z
                      ↑
    y ──→ [3*y]  ──→─┘
    ```
    
    `z.backward()` walks this graph **right to left**, computing each gradient:
    
    ```
    Starting gradient: dz/dz = 1        # always starts at 1
    
    Right branch:  d(3y)/dy = 3         → y.grad = 3
    Left branch:   d(x²)/dx = 2x = 4   → x.grad = 4
    ```
    
    Each branch is **independent**, so they don't affect each other.
    
    ---
    
    ## Step 4: Verify it
    
    ```python
    print(x.grad)  # tensor(4.)  ← 2*x = 2*2
    print(y.grad)  # tensor(3.)  ← constant, always 3 regardless of y's value
    ```
    
    ---
    
    ## The Core Intuition
    
    `x.grad = 4` means:
    > "If I nudge x slightly upward from 2, z will increase by about 4× that nudge"
    
    This is exactly what gradient descent uses to decide **how much and which direction** to adjust each weight.

    ## What is `requires_grad=True`?
    
    Think of it as a **flag that tells PyTorch: "Please remember everything I do with this tensor so I can compute gradients later."**
    
    ### Simple Example
    
    ```python
    import torch
    
    # WITHOUT gradient tracking
    a = torch.tensor(3.0)
    b = a * 2
    b.backward()  # ERROR! Can't compute gradients
    ```
    
    ```python
    # WITH gradient tracking
    x = torch.tensor(3.0, requires_grad=True)
    y = x * 2      # PyTorch remembers: "y came from x"
    y.backward()   # Now we can compute: dy/dx = 2
    print(x.grad)  # tensor(2.)
    ```
    
    ### Why Do We Need It?
    
    **Neural networks need to learn** - they adjust weights to reduce errors. To know *which direction* to adjust, they need gradients (derivatives).
    
    ```
    Weight → Prediction → Loss
             ↑                    ↑
             └── gradient tells us how to adjust ──┘
    ```
    
    ### When to Use It
    
    | Situation | `requires_grad` |
    |-----------|-----------------|
    | Model weights (learnable params) | `True` |
    | Input data | `False` (usually) |
    | Inference/predictions | `False` |
    
    ### Visual Way to Think About It
    
    ```
    requires_grad=False:  x ──► y ──► z   (just compute values, forget how)
                                          
    requires_grad=True:   x ──► y ──► z   (compute values AND remember path)
                           ↑         ↑
                           └─────────┘ (can trace back to compute gradients)
    ```
    
    ### Key Insight
    
    Without `requires_grad=True`, PyTorch treats tensors as **plain numbers** - it computes results but doesn't remember the computational history. With it enabled, PyTorch builds a **computational graph** that lets it work backwards and compute derivatives automatically.
    
    This is the foundation of how all neural networks learn in PyTorch!

    ```python
    z = x**2 + 3*y
    ```

    ### The Math (What PyTorch Does)
    
    **Goal**: Find ∂z/∂x (partial derivative of z with respect to x)
    
    ```
    z = x² + 3y
    ```
    
    **Step 1**: Treat y as a constant (we're differentiating with respect to x only)
    
    ```
    ∂z/∂x = ∂(x²)/∂x + ∂(3y)/∂x
    ```
    
    **Step 2**: Apply derivative rules
    - Derivative of x² is 2x
    - Derivative of 3y (constant) is 0
    
    ```
    ∂z/∂x = 2x + 0 = 2x
    ```
    
    **Step 3**: Plug in x = 2
    
    ```
    ∂z/∂x = 2 × 2 = 4
    ```
    
    ### The Result
    
    ```python
    print(x.grad)  # tensor(4.)
    print(y.grad)  # tensor(3.)  ← ∂z/∂y = 3
    ```
    
    ### Visual Breakdown
    
    ```
    z = x² + 3y
        ───   ───
         ↓     ↓
       2x=4   3 (constant w.r.t. x)
         ↓
       x.grad = 4
    ```
    
    ### Key Insight
    
    The `+ 3*y` part **doesn't affect x.grad** because y is independent of x. PyTorch computes partial derivatives — it asks "if I wiggle x slightly while keeping y fixed, how much does z change?"
    
    This is exactly what we need for neural networks, where each weight has its own gradient!
